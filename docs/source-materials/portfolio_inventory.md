### ⭐ 후보 #1 — 주문 완료 처리 Command/Provider 패턴 리팩토링

| | |
|---|---|
| 티켓 | `INNO-4911` |
| 기간 | 2025-02 ~ 2025-09 (**7개월**) |
| 커밋 수 | **44** |
| 도메인 | 주문 |

**작업 내용 (요약)**
- 주문 완료 처리의 거대한 단일 서비스 로직을 **Command Pattern + Provider Pattern**으로 분해
  - Provider: Command 생성 조건 판단 / 데이터 준비 책임
  - Command: Provider가 준비한 데이터로 실제 작업 수행
- 외부 API 연동을 **Spring Cloud OpenFeign**으로 마이그레이션 (KIPS, 교보e캐시, 교보캐시, e교환권, 통합포인트, 예치금, SAM 이용권, DGV, 위버스 쿠폰 등)
- **RequestContextHolder**로 Request Scope 내 BO 인증 토큰 캐싱 — 동일 트랜잭션 내 중복 토큰 발급 방지
- 정방향/역방향(롤백) Command 분리 — 예외 발생 시 후처리/보상 트랜잭션 명확화
- 1차 캐시(JPA) 문제 해결 + Null-safety 강화

**왜 이게 강력한가**
디자인 패턴 적용 → 외부 통합 → 성능 최적화 → 장애 대응(롤백)까지 풀 사이클을 보여주는 **단일 케이스**. "거대한 레거시를 어떻게 점진적으로 풀었는가" 서사로 쓰기 좋음.

- 본인인증 체크 로직 간소화 — 회원 토큰만으로 ROLE 확인, 외부 API 호출 제거 *(운영 반영)*

**왜 이게 강력한가**
헤드라인은 **"`Box` → typed record 마이그레이션 + AOP 인증 일원화"** + 부 헤드라인 **"캐시 적용 영역 판단(단일키는 도입, 다중키 일부는 보류)"**. 도메인 모델링이 토대가 되어 AOP·캐시·비즈니스 규칙 응집을 모두 가능케 한 사례. 운영 수치(BO 세션키 호출 피크 148K/일 → 0)로 단일키 캐시 적용 판단의 정당성을 데이터로 입증.

### ⭐ 후보 #6 — 기프트카드 발급: 배치 폴링 → Spring AFTER_COMMIT + 비동기 Kafka 발행 (Outbox-inspired)

| | |
|---|---|
| 티켓 | `INNO-2876` (인프라 시작) → `INNO-3045`/`INNO-3282` (인프라 정착) → `INNO-4147` 등 (응용) |
| 기간 | 2024-04 ~ 2024-11 (인프라 4월·응용 단계적) |
| 본인 기여 | **인프라 AtoZ 단독 + 기프트카드 응용 주도**. 배송 도메인은 팀 작업(본인 AtoZ 아님)이라 도메인 지식으로 활용 |
| 도메인 | 공통 (이벤트 인프라) + 주문/기프트카드 |

**작업 내용 (요약)**
- **Spring 이벤트 + Kafka 발행 인프라 단독 구축** (2024-04-09 ~ 2024-04-11, 단 3일):
  - `EventMessage` (제네릭 페이로드: 이벤트구분코드 + 채널 + 순번 + targetId + JSON content)
  - `EventPort` UseCase 인터페이스 (`saveAndPublish`, `publishForApplication` 두 메서드)
  - `EventService` (감사 로그 INSERT + ApplicationEventPublisher publish를 같은 트랜잭션에서)
  - `EventPublishAdapter` (Spring `ApplicationEventPublisher` 어댑터)
  - `EventPersistenceAdapter` + `OrderEventHistoryEntity` (감사 로그 테이블)
  - `KafkaEventListener` (`@TransactionalEventListener` AFTER_COMMIT + `@Async` — commit 후 Kafka 발행)
  - `EvntTypeDvsnCode` enum (REQUEST_CREATE_DELIVERY_REQUEST, REQUEST_UPDATE_DELIVERY_POINT, REQUEST_CREATE_WEVERSE_DELIVERY_REQUEST, **REQUEST_CREATE_GIFTCARD_DELIVERY_REQUEST**)
- **기프트카드 발급 적용**: 주문 완료 시 `DeliveryInfoCommandService.requestDeliveryInfoByOnlineOrder`에서 `isGiftCard` 분기 → `eventPort.saveAndPublish(...REQUEST_CREATE_GIFTCARD_DELIVERY_REQUEST...)` 호출 → AFTER_COMMIT 후 KafkaEventListener가 Kafka 발행 → 외부 컨슈머가 받아 즉시 KIPS 발급 호출
- **별도 배치 서버의 기프트카드 발급 폴링 프로세스 종료** (해당 배치는 fo-order-api가 아닌 외부 배치 서버에서 동작했음)
- **결제와 발급 트랜잭션 명시 분리**: 결제 트랜잭션은 감사 로그 INSERT + ApplicationEvent 발행까지 동기, Kafka 발행은 commit 후 비동기

**임팩트**
- 발급 지연: **최대 15분 → 즉시**
- 별도 배치 서버 의존성 제거
- Rollback ghost message 차단: AFTER_COMMIT 보장으로 rollback된 트랜잭션 메시지가 외부로 안 나감

**한계 (정직한 표기)**
- ⚠️ **순수 Transactional Outbox 패턴은 아님** — `event_history`는 감사 로그이고 CDC/폴링 워커가 read-back으로 발행하지 않음
- ⚠️ JVM 다운 시 손실 가능 — commit 직후 KafkaEventListener 실행 전 인스턴스 다운 시 in-memory EventMessage 사라짐
- 발전 방향: CDC(Debezium)나 폴링 워커가 `event_history` 모니터링하면 진짜 Outbox로 발전 가능

**왜 이게 강력한가**
- 0→1 인프라(Spring 이벤트 + Kafka 비동기 발행) 직접 구축 + 응용까지 = "**아키텍처 패턴 이해 + 적용**"
- **패턴 이름과 실제 구현의 차이를 정확히 인식** — 본 구조가 "Outbox-inspired"이지 순수 Outbox 아님을 정직하게 말할 수 있는 게 시니어 신호
- 팀 작업(배송)과 본인 단독 AtoZ(기프트카드) 명확 구분 — 면접관에게 정직하면서 본인 기여 명확
- 실시간성 임팩트 수치(15분 → 즉시) + 시스템 의존성 제거(배치 서버 종료) = 비기술/기술 모두 어필
- 후보 #1(Command/Provider 보상 패턴)과 사고가 연결됨 — 둘 다 "이벤트 → 처리 → 보상/재시도" 계열
